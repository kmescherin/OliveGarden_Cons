# Техническое задание: электронный консьерж ЖК

## 1. Цели и границы продукта

**Цель:** единое веб-приложение (PWA) для жильцов ЖК в Турции: актуальная информация о комплексе, правилах, режиме социальных зон и составе правления; регистрация с модерацией правлением; заявки на бытовые сервисы; чат-ассистент с RAG по утверждённым материалам (правила, регламенты, FAQ).

**Вне первой версии (этап 2):** нативные приложения, онлайн-оплата, прямые API поставщиков воды, полноценный мессенджер между жильцами.

**Стек:**

- Клиент: Next.js (App Router), TypeScript, Tailwind CSS, shadcn/ui, Magic UI, PWA.
- Бэкенд и данные: Supabase (локально и в облаке): PostgreSQL, Auth, Storage, Row Level Security, Edge Functions.
- ИИ: RAG на pgvector; эмбеддинги и LLM только на сервере (Edge Functions / Route Handlers).

---

## 2. Роли

| Роль | Описание |
|------|----------|
| `guest` | Публичная витрина без персональных данных. |
| `resident_pending` | Зарегистрирован, ожидает подтверждения правления. |
| `resident` | Подтверждённый жилец: сервисы, RAG, личный кабинет. |
| `board` | Модерация регистраций, контент, заявки (по политике). |
| `admin` | Полный доступ; на старте может совпадать с `board`. |

**Поток:** регистрация → `profiles.approval_status = pending` → решение правления → `approved` / `rejected` → RLS открывает функционал для одобренных.

## 3. Модули (M1–M7)

### M1 — Публичный сайт и PWA

- Страницы о комплексе, контакты; установка как PWA.
- manifest, иконки, service worker (кэш статики).
- Адаптив; языки TR / RU (опционально EN) через i18n.

### M2 — Аутентификация и профиль

- Supabase Auth (email/password и/или magic link).
- Профиль: ФИО, блок/квартира, телефон, `approval_status`, `reviewed_by`, `reviewed_at`.
- Страницы: вход, регистрация, ожидание подтверждения, профиль.

### M3 — Модерация правлением

- Список заявок на доступ; одобрить / отклонить с комментарием.
- `audit_log`: кто, когда, какое действие (в payload: статус и текст комментария).
- **Реализовано в приложении:** страницы `/{locale}/board/moderation` (роли `board` / `admin`), RPC `moderate_profile`, журнал модерации в UI; RLS на чтение `audit_log` для правления; исправление рекурсии RLS на `user_roles` (функция `user_has_staff_role`).
- **Админ-панель:** `/{locale}/admin` и `/{locale}/admin/users` — только роль `admin`; список пользователей Auth, выдача/снятие ролей `board` и `admin` (server actions + `service_role`). Первого администратора один раз задают скриптом на сервере или SQL (см. README).
- Страница `/{locale}/admin-required`, если вошли без роли `admin`.

### M4 — Контент

- Правила, часы социальных зон, объявления, контакты правления.
- Изображения в Supabase Storage.
- `visibility`: публично / только жильцы; RLS.

### M5 — Сервисы

- Справочник `service_types`; заявки с описанием, фото, временем, статусом (`new`, `in_progress`, `done`, `cancelled`).
- «Мои заявки»; для правления — очередь и смена статуса.

### M6 — RAG-агент

- Корпус: загруженные документы, контент M4, FAQ.
- Ingest: Storage → чанкинг → эмбеддинг → `document_chunks`.
- Диалог: top-k по вектору → LLM с контекстом и источниками.
- Доступ: только одобренные жильцы; rate limit; минимизация ПДн в промптах.

### M7 — Уведомления (1.5)

- Email; позже Web Push.

## 4. Нефункциональные требования

- RLS на всех чувствительных таблицах; `service_role` только на сервере.
- Пагинация, lazy-load изображений, кэш публичного контента где уместно.
- Логи Edge Functions; при росте — Sentry.
- Резервное копирование: бэкапы Supabase + экспорт Storage.

---

## 5. KVKK (Турция)

- В политике конфиденциальности указать обработку данных и использование ИИ.
- В LLM не передавать персональные данные, не относящиеся к вопросу о ЖК.
- Дисклеймер в UI чата для юридически значимых тем.
